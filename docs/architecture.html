🏛 Architecture Overview

The project follows a Layered (N-Tier) Architecture split into clear, single-responsibility layers. HTTP concerns never mix with business logic, and business logic never touches data definitions.

🔄 Request Flow — Convert Endpoint

Every conversion request follows the same 6-step path from the client through to the JSON response.

1
Client sends HTTP GET

GET /api/conversions/convert?value=100&from=celsius&to=fahrenheit
Any HTTP client — browser, curl, Postman, Invoke-RestMethod.

2
ConversionsController.Convert()

ASP.NET Core binds query params to method arguments. The controller checks that from and to are not empty, then delegates entirely to IConversionService.

3
ConversionService — Unit Lookup

Calls UnitRegistry.TryGetUnit(from) and TryGetUnit(to). If either key is unknown, a ConversionException is thrown immediately.

4
ConversionService — Category Validation

Checks that both units share the same category string. Mixing length and weight is rejected with a descriptive error message.

5
Two-Step Conversion Math

The conversion goes through the category's base unit as an intermediate step:

result = toUnit.FromBase( fromUnit.ToBase( value ) )

Example: 100 °C → °F

// Step 1 — celsius.ToBase: identity (celsius IS the base)
double baseValue = 100.0 * 1.0;  // = 100.0 °C

// Step 2 — fahrenheit.FromBase: apply the affine formula
double result = baseValue * (9.0 / 5.0) + 32.0;  // = 212.0 °F
6
Controller returns JSON response

On success → 200 OK with a ConversionResponse object.
On ConversionException → 400 Bad Request with a ProblemDetails body.

🔗 Dependency Injection Wiring

All wiring happens in Program.cs. No class manually creates its own dependencies.

// UnitRegistry built once — unit definitions are immutable
builder.Services.AddSingleton<UnitRegistry>();

// New service instance per HTTP request (safe, stateless)
builder.Services.AddScoped<IConversionService, ConversionService>();

Singleton — UnitRegistry

Unit definitions never change at runtime. Building the dictionary once and reusing it avoids repeated allocations and is safe for concurrent reads.

Scoped — ConversionService

Stateless per-request. Scoped lifetime is the ASP.NET convention for services that don't hold state and don't need to share state between requests.

💡 Why We Chose This Architecture

Separation of Concerns

Each layer has exactly one job. The controller never does math; the service never formats JSON; the registry never knows about HTTP. This makes each file independently readable and testable.

Testability

ConversionService depends on UnitRegistry (a concrete singleton) and exposes an IConversionService interface. Unit tests instantiate the service directly; integration tests use WebApplicationFactory — no mocking frameworks needed.

Scalability

The system currently hardcodes units. To load them from a database, you only modify UnitRegistry — zero changes to the service, controller, or tests. The interface boundary protects the rest of the codebase.

Extensibility

Adding a new unit = one yield return line in BuildUnits(). Adding a new category = same, just with a new category string. No enums to update, no switch statements to extend.

📐 Conversion Math Design

Why a Base-Unit Strategy?

If you have N units in a category, a naive approach needs N × (N-1) conversion functions. The base-unit strategy needs only 2 × N functions — one ToBase and one FromBase per unit.

10 units naive  = 90 functions
10 units base   = 20 functions

Linear vs Affine

Linear units (length, weight, volume…) only need a scale factor. Stored as a double and wrapped by the Linear() helper.

Affine units (temperature) need scale + offset. Stored as explicit Func<double, double> lambdas. Same interface, different implementation — the caller never needs to know.

Naive Approach — Direct Pair Conversion

With 4 length units (meters, feet, inches, kilometers) you'd need a dedicated function for every possible pair:

// 12 functions just for 4 units — N × (N-1) = 4 × 3 = 12
meters    → feet         ✓
meters    → inches       ✓
meters    → kilometers   ✓
feet      → meters       ✓
feet      → inches       ✓
feet      → kilometers   ✓
inches    → meters       ✓
inches    → feet         ✓
inches    → kilometers   ✓
kilometers → meters      ✓
kilometers → feet        ✓
kilometers → inches      ✓

Adding a 5th unit means writing 4 new functions (one to each existing unit). At 20 units you'd have 380 functions.

Base-Unit Strategy — Everything Goes Through the Middleman

Every unit only needs to know how to talk to meters (the base). That's always exactly 2 functions per unit, no matter how many other units exist:

// 8 functions for 4 units — 2 × N = 2 × 4 = 8
meters     → ToBase (× 1.0)      FromBase (÷ 1.0)
feet       → ToBase (× 0.3048)   FromBase (÷ 0.3048)
inches     → ToBase (× 0.0254)   FromBase (÷ 0.0254)
kilometers → ToBase (× 1000)     FromBase (÷ 1000)

To convert feet → inches there is no direct function. It routes through meters automatically:

feet → (ToBase) → meters → (FromBase) → inches
1 ft  →  0.3048 m  →  11.97 inches

Adding a new unit always costs exactly 2 functions, regardless of how many units already exist.

🛡 Error Handling Strategy

Scenario Where Detected Exception HTTP Response
Unknown unit key ConversionService ConversionException 400 Bad Request
Cross-category conversion ConversionService ConversionException 400 Bad Request
Missing query parameter Controller — 400 Bad Request
Invalid number format ASP.NET Core model binding — 400 Bad Request

All error responses use the standard RFC 7807 ProblemDetails format, so clients get a consistent, machine-readable error body.
